# 中国 (CN, +86) AWS SMS 发送专题研究报告

> 调研日期：2026年4月7日
> 数据来源：AWS 官方文档、工信部官网、中国三大运营商资料

## 一、AWS 上的中国 SMS 能力总览

| 项目 | 详情 |
|------|------|
| 国家代码 | +86 |
| 短码 (Short Code) | **支持** |
| 长码 (Long Code) | 不支持 |
| Sender ID | 不支持 |
| Toll-free | 不支持 |
| 双向 SMS | **不支持** |
| 消息拼接 | 支持 |
| MMS | 不支持 |
| 国际发送 | 支持 |
| **模板注册** | **必须（AWS 上唯一要求模板注册的国家）** |

**核心限制：** 中国是 AWS 上唯一要求消息模板预注册（Template Registration）的国家。未经注册的模板消息将被直接拦截，无法送达。此外，Transactional 类型消息严禁包含任何营销/推广内容。

## 二、模板注册（Template Registration）详细流程

### 2.1 为什么必须注册模板

中国运营商对 A2P SMS 实行严格的模板审核机制。所有通过 AWS 发往中国 +86 号码的短信，必须使用预先注册并获批的模板。未注册模板的消息将被运营商侧直接拦截，不会产生送达记录。

### 2.2 注册方式

通过 **AWS Support Case** 提交模板注册申请。

### 2.3 Support Case 填写指南

**访问地址：**
```
https://console.aws.amazon.com/support/home#/case/create?issueType=service-limit-increase
```

| 字段 | 填写值 |
|------|-------|
| Issue Type | Service Limit Increase |
| Service | Service Quotas |
| Category | AWS End User Messaging SMS (Pinpoint) |
| Severity | General Limits |
| Region | 你使用的 AWS 区域（如 us-east-1） |
| Resource Type | General Limits |
| Limit Type | AWS End User Messaging SMS (Pinpoint) |
| Limit Increase Value | **1** |

### 2.4 Case Description 模板

```
Subject: SMS Template Registration Request - China (CN, +86)

We are requesting SMS template registration for sending messages to
China (+86).

Company Information:
- Company Name: [公司法定名称]
- Company Address: [公司地址]
- Contact Email: [联系邮箱]
- Contact Phone: [联系电话]
- Website: [公司网站]

Message Type: Transactional (OTP / Notifications)

Target Country: China (+86)
Target Carriers: China Mobile, China Unicom, China Telecom

Expected Monthly Volume: [预估月发送量，如 100,000 messages]
Expected Messages per User per Month: [每月每用户发送数量，如 5]

SMS Templates for Registration:

Template 1 - OTP Verification:
【{Brand}】您的验证码是：{code}，有效期{minutes}分钟。请勿将验证码告知他人。

Template 2 - Transaction Notification:
【{Brand}】您的订单{orderID}已确认，金额：￥{amount}元。详情请登录官网查看。

Template 3 - Security Alert:
【{Brand}】您的账户在新设备登录，如非本人操作，请立即修改密码。

Opt-in Process:
Users register on our website/app and provide their Chinese mobile number.
During registration, they check a consent box stating: "我同意接收来自
[Brand] 的短信验证码和账户通知。"

Opt-out Mechanism:
Users can manage SMS preferences through their account settings page
or contact customer service at support@brand.com to opt out.

We confirm that:
1. All templates are for transactional purposes only (no marketing content)
2. We have obtained user consent for sending these messages
3. We will comply with Chinese telecommunications regulations

Please provide the registration form and estimated timeline.
```

### 2.5 注册时间线

| 阶段 | 预计时间 |
|------|---------|
| AWS 确认收到 | 24小时内 |
| AWS 提供国家专属注册表单 | 1-3个工作日 |
| 填写并提交注册表格 | 取决于你 |
| 模板审核 | 1-4周 |
| 模板上线 | 审核通过后即时生效 |
| **总计** | **2-6周** |

### 2.6 模板注册要求

1. **模板必须包含签名**：格式为 `【品牌名】`，使用中文方括号
2. **变量用占位符标识**：如 `{code}`、`{orderID}`、`{amount}` 等
3. **每种消息类型需单独注册**：OTP 模板、通知模板、警报模板等需分别注册
4. **Transactional 类型严禁营销内容**：不得包含任何推广、促销、打折、优惠等信息
5. **模板内容需与申报用途一致**：OTP 模板只能发 OTP，不能混用

### 2.7 模板审核失败常见原因

| 原因 | 说明 |
|------|------|
| 含营销内容 | Transactional 模板中包含促销、优惠信息 |
| 缺少签名 | 模板未包含 `【品牌名】` 签名 |
| 签名未备案 | 品牌签名未在运营商处备案 |
| 内容违规 | 涉及赌博、色情、政治等敏感内容 |
| 变量过多 | 模板中变量占比超过固定文本 |
| 用途不匹配 | 实际发送内容与注册模板不符 |

## 三、中国电信监管

### 3.1 监管机构

**工业和信息化部（MIIT）** 是中国电信行业的核心监管机构，负责制定和执行通信管理政策。

### 3.2 核心法规

| 法规 | 主要内容 |
|------|---------|
| 《通信短信息和语音呼叫服务管理规定》（2020年修订） | 规范商业短信发送，要求事先同意 |
| 《反电信网络诈骗法》（2022年） | 严打电信诈骗，加强短信内容监管 |
| 《电信条例》 | 电信服务基本管理框架 |
| 《互联网信息服务管理办法》 | 互联网信息内容管理 |

### 3.3 A2P SMS 核心监管要求

- **短信签名备案制**：所有 A2P 短信必须包含已备案的签名（`【品牌名】`）
- **模板审核制**：短信内容须使用审核通过的模板
- **发送频率限制**：运营商对同一号码的发送频率有限制（通常同号码每分钟不超过一定条数）
- **发送时间限制**：营销短信限 08:00-21:00
- **退订机制**：商业短信须提供退订方式
- **实名制**：接收方号码均已实名认证

### 3.4 禁止/限制内容

| 类别 | 限制程度 |
|------|---------|
| 未经同意的营销 | **严禁** |
| 赌博/博彩 | **严禁** |
| 色情内容 | **严禁** |
| 政治敏感内容 | **严禁** |
| 金融诈骗/钓鱼 | **严禁** |
| 非法集资/传销 | **严禁** |
| 虚假广告 | **严禁** |
| 烟草广告 | **严禁** |
| 处方药广告 | **严禁**（需特殊审批） |

### 3.5 发送时间限制

- **事务性消息（OTP、通知）**：24小时均可发送
- **营销/推广消息**：限 08:00-21:00 CST（UTC+8）
- 国定假日期间运营商可能加强内容过滤

## 四、中国数据保护法

### 4.1 核心法律

中国有三部核心数据保护法律，构成完整的数据保护体系：

| 法律 | 生效时间 | 核心关注 |
|------|---------|---------|
| 《个人信息保护法》（PIPL） | 2021年11月 | 个人信息处理规则（类似 GDPR） |
| 《数据安全法》（DSL） | 2021年9月 | 数据分类分级保护 |
| 《网络安全法》（CSL） | 2017年6月 | 网络运营者安全义务 |

### 4.2 对 SMS 业务的关键要求

| 要求 | 详情 |
|------|------|
| **用户同意** | 必须获得明确的、知情的同意才能发送 SMS |
| **最小必要原则** | 仅收集和处理必要的个人信息（手机号码） |
| **数据本地化** | 在中国境内收集的个人信息，原则上应在境内存储 |
| **跨境传输** | 个人信息出境需通过安全评估、标准合同或认证 |
| **个人权利** | 用户有权查阅、复制、更正、删除其个人信息 |
| **数据保留** | 不应超过实现处理目的所必需的最短期限 |
| **安全保护** | 采取加密、去标识化等技术措施保护个人信息安全 |

### 4.3 违规处罚

| 处罚类型 | 详情 |
|---------|------|
| 一般违规 | 责令改正、没收违法所得、罚款最高 100 万元人民币 |
| 情节严重 | 罚款最高 5,000 万元人民币或上一年度营业额的 **5%** |
| 直接责任人 | 个人罚款 10-100 万元人民币，可禁止担任高管 |
| 行政处罚 | 暂停相关业务、停业整顿、吊销业务许可证 |
| 刑事责任 | 侵犯公民个人信息罪，最高 **7年有期徒刑** |

## 五、运营商与号码格式

### 5.1 三大运营商

| 运营商 | 英文名 | 市场份额（约） | 号段示例 |
|--------|--------|-------------|---------|
| 中国移动 | China Mobile | ~57% | 134-139, 150-152, 157-159, 182-184, 187-188, 195, 198 |
| 中国联通 | China Unicom | ~20% | 130-132, 155-156, 185-186, 166, 196 |
| 中国电信 | China Telecom | ~23% | 133, 149, 153, 173, 177-178, 180-181, 189, 191, 193, 199 |

### 5.2 号码格式

#### 基本结构
- 国家代码：+86
- 手机号码长度：11位（不含国家代码）
- 格式：+86 1XX XXXX XXXX

#### AWS SMS 正确格式
```
+8613800138000     ← 中国移动示例
+8615500155000     ← 中国联通示例
+8618900189000     ← 中国电信示例
```

> **重要：** 发送时必须使用 E.164 格式，即 `+86` 后跟 11 位手机号码，中间不加空格或横线。固定电话号码不可接收短信。

#### 号码验证规则
- 必须以 `1` 开头（紧跟 `+86` 之后）
- 总长度为 13 位（含国家代码 `86`）
- 不包含前导 `0`（国内拨打手机不加 0）

### 5.3 运营商技术细节

- 三大运营商均支持 A2P SMS
- 运营商有严格的内容过滤系统
- 高频发送同一内容会触发限流
- 同一号码短时间内接收过多短信会被保护性拦截
- 支持号码携带（携号转网，2019年全国实施）
- 虚拟运营商号码（170/171 等）可能投递率较低

## 六、不支持双向 SMS 的替代退出方案

### 6.1 问题

AWS 上中国不支持双向 SMS，用户无法回复 STOP 退订。必须提供替代 opt-out 机制。

### 6.2 替代方案

#### 方案一：App/网站内设置（推荐）

在 App 或用户中心提供短信偏好管理：
- 用户登录后在"设置 > 通知偏好"中关闭短信通知
- 保留退出记录作为合规证据

#### 方案二：客服渠道退出

```
如需退订，请联系客服：support@brand.com 或拨打 400-XXX-XXXX
```

#### 方案三：短信中附带退出链接

```
【Brand】退订请访问：https://brand.com/unsub/{TOKEN}
```

> **注意：** 中文短信每条仅 70 字符（Unicode 编码），加入退出链接会大幅增加消息段数和成本，需权衡。

### 6.3 最佳实践

1. OTP/安全类消息通常无需退出机制（用户主动触发）
2. 通知类消息建议在 App/网站内提供管理入口
3. 保留所有 opt-in 和 opt-out 记录
4. 退出后24小时内停止发送

## 七、消息模板范例（中文）

### 7.1 字符编码说明

| 编码 | 单条上限 | 拼接每段 | 何时触发 |
|------|---------|---------|---------|
| GSM-7 | 160字符 | 153字符 | 仅基本拉丁字符 |
| UCS-2 (Unicode) | **70字符** | **67字符** | **含中文字符时自动触发** |

**中文消息强制使用 Unicode 编码：**

由于中文字符不在 GSM-7 字符集中，所有包含中文的消息将自动使用 UCS-2 (Unicode) 编码。这意味着：

| 对比项 | GSM-7（纯英文） | UCS-2（含中文） |
|--------|---------------|----------------|
| 单条最大字符数 | 160 | **70** |
| 拼接每段字符数 | 153 | **67** |
| 70字符消息段数 | 1段 | **1段** |
| 140字符消息段数 | 1段 | **3段**（67+67+6） |

> **成本影响巨大：** 一条 140 字符的中文消息需要 3 个消息段，费用是单条的 3 倍。因此中文短信必须严格控制字数，尽量在 70 字符以内。

### 7.2 OTP 验证码模板

**模板（Unicode，70字符/条）：**
```
【Brand】您的验证码是：385291，有效期5分钟。请勿告知他人。
```
（30字符，1条消息）

**纯英文版（如面向在华外籍用户，GSM-7，160字符/条）：**
```
[Brand] Your verification code is: 385291. Valid for 5 minutes.
Do not share this code.
```
（80字符，1条消息）

### 7.3 交易通知模板

**订单确认：**
```
【Brand】您的订单{orderID}已确认，金额：￥{amount}元。预计{date}送达。
```
（约40字符，1条消息）

**发货通知：**
```
【Brand】您的订单{orderID}已发货，快递单号：{trackingNo}。
```
（约35字符，1条消息）

**签收确认：**
```
【Brand】您的包裹已签收。如有问题请联系客服400-XXX-XXXX。
```
（约32字符，1条消息）

**付款确认：**
```
【Brand】收到您的付款￥{amount}元（订单{orderID}）。感谢购买。
```
（约38字符，1条消息）

### 7.4 安全告警模板

**新设备登录：**
```
【Brand】您的账户在新设备登录。如非本人操作，请立即修改密码。
```
（约30字符，1条消息）

**异常操作：**
```
【Brand】检测到您的账户异常操作，已临时锁定。请联系客服解锁。
```
（约31字符，1条消息）

### 7.5 模板变量说明

| 变量 | 说明 | 示例 |
|------|------|------|
| `{code}` | 验证码 | 385291 |
| `{orderID}` | 订单号 | CN20260407001 |
| `{amount}` | 金额 | 299.00 |
| `{date}` | 日期 | 4月10日 |
| `{trackingNo}` | 快递单号 | SF1234567890 |
| `{Brand}` | 品牌签名 | 按实际品牌名 |

### 7.6 注意事项汇总

| 要点 | 说明 |
|------|------|
| 语言 | 中文（简体） |
| 编码 | **强制 Unicode（UCS-2）**，每条仅 70 字符 |
| 签名 | 必须包含 `【品牌名】`，使用中文方括号 |
| 模板注册 | **必须预注册，未注册直接拦截** |
| 字符控制 | 严格控制在 70 字符以内避免拼接 |
| 营销禁止 | Transactional 模板严禁包含任何营销内容 |
| 货币符号 | 使用 `￥` 或 `元`（`￥` 为 Unicode 字符） |
| 变量占比 | 模板中固定文本应多于变量部分 |

## 八、定价信息

### 8.1 AWS 定价

AWS 未在公开文档直接列出中国每条费率，需通过：
1. 下载 AWS 定价 CSV：`aws.amazon.com/end-user-messaging/pricing/`
2. 联系 AWS 销售获取报价
3. 登录 AWS 控制台查看

**已知费用结构：**
- 每条消息费：需参考 AWS 定价 CSV（中国单价通常较高）
- 默认消费阈值：$1.00 USD/月（需申请提高）
- 模板注册本身通常不额外收费

### 8.2 成本优化建议

1. **严格控制字数在 70 字符以内** — 超过 70 字符将按 2 条计费（67+剩余），成本翻倍
2. **使用简洁的模板** — 去掉不必要的客套话，保留核心信息
3. **尽量用数字而非中文数字** — `5` 比 `五` 少 0 个额外字节，但每个字符在 UCS-2 下均占 2 字节
4. **URL 尽量短** — 如需包含链接，使用品牌短域名
5. **维护号码列表** — 定期清理无效号码、空号、虚拟号
6. **分时段批量发送** — 避免瞬时高峰触发运营商限流

## 九、合规清单

在中国发送 SMS 前，确保完成：

- [ ] 所有消息模板已通过 AWS 注册审核
- [ ] 获取所有接收者的明确 opt-in 同意
- [ ] 实施有效的 opt-out 机制（App 内设置/客服渠道，因 AWS 不支持双向）
- [ ] 模板包含品牌签名 `【品牌名】`
- [ ] Transactional 消息不含任何营销/推广内容
- [ ] 消息内容不涉及禁止/敏感内容
- [ ] 遵守《个人信息保护法》数据处理要求
- [ ] 跨境数据传输已完成安全评估或标准合同
- [ ] 正确格式化号码（+86 + 11位手机号码）
- [ ] 每条消息控制在 70 字符以内（Unicode 编码）
- [ ] 发送频率符合运营商限制
- [ ] 保留 opt-in 同意记录和发送日志
- [ ] 准备备用通信渠道（微信/邮件）
- [ ] 定期清理号码列表，移除无效和已退订号码
- [ ] 响应个人信息删除请求（15个工作日内）

## 十、风险评估

| 风险 | 严重程度 | 缓解措施 |
|------|---------|---------|
| 模板注册被拒 | **极高** | 确保内容合规，Transactional 不含营销内容，提供完整材料 |
| 未注册模板消息被拦截 | **极高** | 上线前确认所有模板已注册通过，建立模板管理流程 |
| 运营商内容过滤 | 高 | 使用已注册模板，避免敏感词汇，保持发送频率稳定 |
| Unicode 编码导致成本剧增 | 高 | 严格控制每条消息在 70 字符以内 |
| 数据保护违规（PIPL） | 高 | 严格遵守 PIPL，完成跨境数据传输评估 |
| 运营商限流 | 中 | 控制发送速率，分批发送，避免瞬时高峰 |
| 号码格式错误 | 中 | 实施严格的号码格式验证（+86 + 11位） |
| 虚拟运营商号码投递率低 | 中 | 识别虚拟运营商号段，考虑备用通知渠道 |
| 签名备案问题 | 中 | 提前确认品牌签名是否已在运营商处备案 |
