# 韩国 (KR, +82) AWS SMS 发送专题研究报告

> 调研日期：2026年4月7日
> 数据来源：AWS 官方文档、韩国放送通信委员会(KCC)资料、韩国信息通信法律资料

---

## 一、AWS 上的韩国 SMS 能力总览

| 项目 | 详情 |
|------|------|
| 国家代码 | +82 |
| 短码 (Short Code) | **不支持** |
| 长码 (Long Code) | **不支持** |
| Sender ID | **不支持** |
| Toll-Free | **不支持** |
| 双向 SMS | **不支持** |
| 消息拼接 | 支持（但不保证） |
| MMS | **不支持** |
| 国际发送 | 支持（仅 best-effort 模式） |

**核心限制：** 韩国是 AWS SMS 功能**最受限**的国家。不支持任何号码类型（短码、长码、Sender ID），不支持双向短信，仅能通过国际路由以 "尽力而为"（best-effort）模式发送。消息可能显示为随机号码，送达率无保证。**强烈建议需要在韩国大量发送 SMS 的企业考虑本地运营商或第三方服务。**

---

## 二、Best-effort 模式详解

### 2.1 什么是 Best-effort 模式

当 AWS 无法为特定国家提供专用号码或注册的 Sender ID 时，消息通过国际 SMS 路由发送。这意味着：

| 特性 | 说明 |
|------|------|
| 发送方式 | 通过国际网关路由 |
| 显示号码 | 可能显示为**随机号码或共享号码** |
| 送达保证 | **无保证**，尽力而为 |
| 送达率 | **不可靠**，可能低于其他国家 |
| 延迟 | 可能较高，无 SLA |
| 运营商过滤 | 国际路由消息更容易被过滤 |
| 回复能力 | **不可回复**（无双向 SMS） |

### 2.2 实际使用中的问题

1. **号码不一致：** 每次发送可能显示不同的随机号码，用户无法识别发送者
2. **运营商过滤：** 韩国运营商对国际 A2P SMS 有严格的过滤机制
3. **高失败率：** 部分消息可能无法送达，无法获得准确的送达报告
4. **用户信任问题：** 随机号码显示可能导致用户将消息视为垃圾短信或诈骗
5. **无法验证身份：** 无法使用品牌名称作为发送者

### 2.3 适用场景

Best-effort 模式**仅适合**以下极有限的场景：

| 场景 | 适用性 | 说明 |
|------|--------|------|
| 偶发性 OTP | 勉强可用 | 仅在没有其他方案时使用 |
| 关键业务通知 | **不推荐** | 送达率无保证 |
| 营销消息 | **不推荐** | 几乎不可行 |
| 大批量发送 | **不推荐** | 可能被运营商封锁 |
| 用户认证 | **不推荐** | 建议使用替代方案 |

## 三、替代方案建议

### 3.1 方案对比总览

| 方案 | 优势 | 劣势 | 适用场景 |
|------|------|------|---------|
| **韩国本地运营商 API** | 最高送达率、品牌显示、合规 | 需本地合同、韩语支持 | 大批量、核心业务 |
| **第三方 CPaaS 平台** | 已建立本地通道、中文/英文支持 | 额外成本、多供应商管理 | 中大批量、快速上线 |
| **KakaoTalk 消息** | 韩国用户渗透率 95%+ | 非 SMS、需注册 Kakao 商家 | 通知、营销、客服 |
| **AWS 国际路由（Best-effort）** | 已有 AWS 基础设施 | 不可靠、随机号码 | 仅偶发 OTP |

### 3.2 方案一：韩国本地运营商 API

**推荐指数：** 高

韩国三大运营商均提供 A2P SMS API 服务：

| 运营商 | 服务名 | 说明 |
|--------|--------|------|
| SK Telecom | T-SMS | 最大运营商，最广覆盖 |
| KT (Korea Telecom) | KT SMS API | 第二大运营商 |
| LG U+ | LG U+ SMS | 第三大运营商 |

**优势：**
- 本地号码或品牌 Sender ID 显示
- 高送达率（99%+）
- 支持韩文 LMS（长消息服务，最多 2,000 字节）
- 支持 MMS
- 合规性最佳

**挑战：**
- 需要韩国法人实体或本地代理
- 合同语言通常为韩语
- 需要本地银行账户进行支付
- 技术文档可能为韩语

### 3.3 方案二：第三方 CPaaS 平台

**推荐指数：** 中高

多个国际 CPaaS 平台在韩国有本地通道：

| 平台 | 韩国支持 | 特点 |
|------|---------|------|
| Infobip | 本地直连 | 在韩国有本地团队 |
| MessageBird | 本地通道 | 亚太区域覆盖好 |
| Solacom / DirectSend | 韩国本地 | 韩国本地 SMS 服务商 |
| NHN Cloud | 韩国本地 | 韩国本地云服务商，SMS API |

> **注意：** 选择第三方平台时，确认其在韩国是否有本地直连通道（Direct Connection），避免通过国际路由中转。

### 3.4 方案三：KakaoTalk 消息（AlimTalk / FriendTalk）

**推荐指数：** 高（对于非 SMS 场景）

KakaoTalk 是韩国最普及的即时通讯应用，月活跃用户超过 4,800 万（韩国人口约 5,100 万），渗透率超过 95%。

| 消息类型 | 说明 | 费用（约） |
|----------|------|-----------|
| **AlimTalk（通知型）** | 模板消息，需审批 | ~15-20 韩元/条 |
| **FriendTalk（好友型）** | 营销消息，需加好友 | ~20-30 韩元/条 |

**AlimTalk 特点：**
- 适用于 OTP、订单确认、配送通知等事务性消息
- 消息模板需提前在 Kakao 审批
- 支持按钮、链接等富媒体元素
- 比 SMS 更便宜
- 如用户未安装 KakaoTalk，可降级为 SMS 发送

**接入方式：**
1. 注册 Kakao Business 账号
2. 创建 Kakao Channel
3. 通过 Kakao 官方 API 或合作伙伴接入

### 3.5 混合方案（推荐）

**最佳实践：** 采用 KakaoTalk 优先、SMS 兜底的混合方案：

```
发送消息
  ├─ 尝试 KakaoTalk AlimTalk
  │   ├─ 成功 → 完成
  │   └─ 失败（用户未安装 Kakao）
  │       ├─ 通过本地运营商 API 发送 SMS
  │       │   ├─ 成功 → 完成
  │       │   └─ 失败 → 记录错误，通知运维
  │       └─ （备选）通过 AWS best-effort 发送
  └─ 异常 → 降级为邮件/推送通知
```

### 4.2 相关法律

| 法律 | 韩文名 | 说明 |
|------|--------|------|
| **信息通信网法** | 정보통신망 이용촉진 및 정보보호 등에 관한 법률 | A2P SMS 的核心规范 |

### 4.3 A2P SMS 监管要求

| 要求 | 详情 |
|------|------|
| **事前同意** | 发送广告性 SMS 前必须获得接收者**事前明确同意** |
| **发送者标识** | 必须在消息中标明发送者名称和联系方式 |
| **退订机制** | 必须提供**免费**的退订方式 |
| **夜间发送限制** | **21:00-08:00 禁止发送广告性 SMS** |
| **广告标识** | 广告性 SMS 必须在消息开头标注 `(광고)` |
| **退订指引** | 广告性 SMS 必须包含免费退订方式说明 |

### 4.4 广告性 SMS 格式要求

韩国法律要求广告性 SMS 必须遵循特定格式：

```
(광고) [발신자명]
[메시지 내용]
무료수신거부 [전화번호 또는 URL]
```

翻译：
```
(广告) [发送者名称]
[消息内容]
免费退订 [电话号码或 URL]
```

### 4.5 禁止/限制内容

| 类别 | 限制程度 |
|------|---------|
| 未经同意的广告 | **禁止**（罚款最高 3,000 万韩元） |
| 深夜广告（21:00-08:00） | **禁止** |
| 欺诈/钓鱼 | **禁止**（刑事处罚） |
| 赌博推广 | **禁止** |
| 成人内容 | 需年龄验证 |
| 金融诈骗 | **禁止**（刑事处罚） |

### 4.6 发送时间限制

| 消息类型 | 允许时段 | 说明 |
|---------|---------|------|
| 广告性 SMS | 08:00-21:00 KST (UTC+9) | 法律强制，违反可罚款 |
| 事务性 SMS | 任何时间 | OTP、安全警报等 |
| 交易通知 | 任何时间 | 订单确认、配送通知等 |

## 五、韩国数据保护法 (PIPA)

### 5.1 法律概述

**PIPA**（Personal Information Protection Act，개인정보 보호법）是韩国核心数据保护法律，于 2011 年生效，2023 年进行了重大修订。PIPA 被认为是亚太地区最严格的数据保护法之一。

### 5.2 对 SMS 业务的关键要求

| 要求 | 详情 |
|------|------|
| **用户同意** | 收集和使用个人信息（包括手机号码）须获得**明确同意** |
| **目的限制** | 个人信息仅可用于收集时告知的目的 |
| **最小收集原则** | 仅收集必要的最少信息 |
| **信息主体权利** | 访问权、更正权、删除权、处理暂停权 |
| **跨境传输** | 向海外传输个人信息需**告知并获得同意** |
| **数据泄露通知** | 发生泄露后 **72 小时内**通知监管机构和受影响个人 |
| **个人信息处理方针** | 必须公开隐私政策 |

### 5.3 跨境数据传输（关键）

由于 AWS 服务器通常位于海外（如 us-east-1），向韩国用户发送 SMS 涉及跨境数据传输：

| 传输方式 | 要求 |
|---------|------|
| 同意 | 告知接收方、传输目的、信息类型，获得同意 |
| 合同 | 与接收方签订数据保护合同 |
| 认证 | 接收方获得 PIPA 认可的数据保护认证 |

> **建议：** 使用 AWS 首尔区域（ap-northeast-2）处理韩国用户数据，减少跨境传输合规风险。但即使如此，SMS 消息仍通过国际路由发送。

### 5.4 违规处罚

| 处罚类型 | 详情 |
|---------|------|
| 行政罚款 | 最高**违法收入的 3%**或**相关销售额的 3%** |
| 刑事处罚 | 最高 **5 年监禁**或 **5,000 万韩元罚款** |
| 个人赔偿 | 受害者可提起民事诉讼 |
| 集体诉讼 | 支持集体诉讼机制 |

## 六、运营商与号码格式

### 6.1 三大运营商

| 运营商 | 英文名 | 市场份额（约） | 备注 |
|--------|--------|-------------|------|
| SK Telecom | SK Telecom | ~44% | 最大运营商 |
| KT | Korea Telecom | ~30% | 第二大，固网最强 |
| LG U+ | LG Uplus | ~21% | 第三大 |
| MVNO | 各类虚拟运营商 | ~5% | 依赖三大网络 |

### 6.2 号码格式

#### 基本结构
- 国家代码：+82
- 手机号码：以 010 开头，共 11 位（含 010）
- 拨打格式：010-XXXX-XXXX

#### 号码格式转换

| 场景 | 格式 | 示例 |
|------|------|------|
| 国内格式 | 010-XXXX-XXXX | 010-1234-5678 |
| **AWS SMS 格式（E.164）** | +82 + 去掉前导 0 | **+8210XXXXXXXX** |

#### 正确的 AWS SMS 号码格式
```
+821012345678     ← 标准手机号码
```

> **关键：** 去掉国内号码的前导 `0`，加上 `+82`。即 `010-1234-5678` 变为 `+821012345678`。

#### 历史号码格式
- 旧格式：011/016/017/018/019 开头（部分仍在使用）
- 现代格式：统一为 010 开头
- 固定电话：02（首尔）、031（京畿道）等，**不可达 SMS**

### 6.3 运营商对国际 SMS 的处理

- 韩国运营商对国际入站 SMS 有严格过滤
- 未注册的国际发送者容易被标记为垃圾短信
- 部分用户可能在手机设置中屏蔽国际短信
- 运营商可能将国际 SMS 的发送号码替换为随机号码

## 七、消息模板范例（韩语）

### 7.1 字符编码说明（极重要）

韩语字符**必须**使用 Unicode (UCS-2) 编码，这是因为韩语字符（韩文音节块 / Hangul）不在 GSM-7 字符集内。

| 编码 | 单条上限 | 拼接每段 | 何时触发 |
|------|---------|---------|---------|
| GSM-7 | 160字符 | 153字符 | **不适用于韩语** |
| **UCS-2 (Unicode)** | **70字符** | **67字符** | **韩语消息始终触发** |

> **关键影响：** 韩语消息每条仅 70 字符，拼接时每段仅 67 字符。这意味着韩语消息的成本约为纯英文消息的 **2.3 倍**（相同内容长度下需要更多消息段）。

### 7.2 韩语字符计数说明

| 字符类型 | UCS-2 中占用 | 说明 |
|---------|-------------|------|
| 韩文音节（가-힣） | 1个字符位 | 每个韩文音节占 1 个 UCS-2 字符位 |
| 英文字母（A-Z, a-z） | 1个字符位 | |
| 数字（0-9） | 1个字符位 | |
| 基本标点 | 1个字符位 | |
| 空格 | 1个字符位 | |

**每条消息预算：** 70 个字符位（含韩文、英文、数字、标点、空格全部加起来）。

### 7.3 OTP 验证码模板

```
[Brand] 인증번호: 385291 (5분 유효)
이 번호를 타인에게 알려주지 마세요.
```
(39字符，1条消息)

**纯英文 OTP（如系统不支持韩语）：**
```
[Brand] Verification code: 385291. Valid for 5 min.
Do not share this code.
```
(71字符，使用 GSM-7 则为 1 条；但如果同条消息中有任何韩文字符则变为 Unicode)

> **建议：** 如果 OTP 消息不包含韩文，可使用纯英文以节省成本（GSM-7, 160字符/条）。韩国用户普遍能阅读英文 OTP。

### 7.4 交易通知模板

**订单确认：**
```
[Brand] 주문 #KR12345 확인됨
금액: ₩150,000
추적: https://brand.com/o/KR12345
```
(62字符，1条 — 但 `₩` 为 Unicode，整条消息为 UCS-2 编码)

**配送通知：**
```
[Brand] 주문 #KR12345 배송시작
배송추적: https://brand.com/t/KR12345
예상도착: 4월 10일
```
(63字符，1条)

**账户安全警报：**
```
[Brand] 새 기기에서 로그인 감지
본인이 아닌 경우:
https://brand.com/security
```
(50字符，1条)

### 7.5 营销消息模板（需事前同意，须遵循韩国广告格式）

**韩国法律要求的广告 SMS 格式：**

```
(광고) [Brand]
특별 할인: 다음 구매 20% 할인!
코드: SAVE20 (4/15까지)
무료수신거부 https://brand.com/unsub/{TOKEN}
```
(69字符，1条)

> **注意：** `(광고)` 标记和 `무료수신거부`（免费退订）是韩国法律强制要求。

**会员积分提醒：**
```
(광고) [Brand]
포인트 500점 4/30 소멸예정
사용: https://brand.com/rewards
무료수신거부 080-XXX-XXXX
```
(65字符，1条)

### 7.6 注意事项汇总

| 要点 | 说明 |
|------|------|
| 语言 | 韩语（한국어），OTP 可用英文 |
| 编码 | 韩语消息**始终**为 UCS-2 (70字符/条) |
| 纯英文 OTP | 建议使用以节省成本（GSM-7, 160字符/条） |
| 广告标识 | 广告 SMS 必须以 `(광고)` 开头 |
| 退订指引 | 广告 SMS 必须包含 `무료수신거부` + 免费退订方式 |
| 夜间禁止 | 21:00-08:00 KST 禁止发送广告 SMS |
| 货币符号 | `₩` 为 Unicode 字符（但韩语消息已是 Unicode，无额外影响） |
| URL 长度 | UCS-2 模式下 URL 也占字符位，需尽量缩短 |

## 八、定价信息

### 8.1 AWS 定价

AWS 未在公开文档直接列出韩国每条费率，需通过：
1. 下载 AWS 定价 CSV：`https://aws.amazon.com/end-user-messaging/pricing/`
2. 联系 AWS 销售获取报价
3. 登录 AWS 控制台查看

**已知费用结构：**
- 无号码月租费（因为不支持任何号码类型）
- 每条消息费：需参考定价 CSV（国际路由通常费率较高）
- 默认消费阈值：$1.00 USD/月（需申请提高）

### 8.2 成本对比（SMS vs KakaoTalk）

| 渠道 | 单条费用（约） | 送达率 | 品牌显示 |
|------|-------------|--------|---------|
| AWS Best-effort SMS | 较高（国际路由） | 不可靠 | 随机号码 |
| 本地运营商 SMS | ~20-30 韩元/条 | 99%+ | 品牌名称 |
| KakaoTalk AlimTalk | ~15-20 韩元/条 | 99%+（安装用户） | 品牌频道 |
| KakaoTalk FriendTalk | ~20-30 韩元/条 | 99%+（好友用户） | 品牌频道 |

> **结论：** 对于韩国市场，AWS Best-effort SMS 在成本和效果上均不如本地方案。

### 8.3 成本优化建议

1. **OTP 用纯英文** — 避免韩文触发 Unicode，使用 GSM-7（160字符/条 vs 70字符/条）
2. **使用 KakaoTalk 优先** — 比 SMS 更便宜且送达率更高
3. **极简消息内容** — Unicode 模式下每字符都珍贵，控制在 70 字符内
4. **评估替代方案** — 对于大批量场景，本地方案总成本可能更低

## 十二、总结与建议

### 12.1 核心结论

**AWS SMS 不适合作为韩国市场的主要 SMS 发送渠道。** 由于不支持任何号码类型、不保证送达率、消息显示为随机号码，AWS 的 best-effort 模式仅适用于极低频的偶发场景。

### 12.2 推荐方案

| 优先级 | 方案 | 说明 |
|--------|------|------|
| **首选** | KakaoTalk AlimTalk | 覆盖 95%+ 韩国用户，成本最低，体验最佳 |
| **次选** | 韩国本地运营商 SMS API | 最高 SMS 送达率，品牌显示 |
| **第三** | 第三方 CPaaS 本地通道 | 快速接入，不需本地合同 |
| **最后** | AWS Best-effort SMS | 仅在上述方案均不可行时使用 |

### 12.3 如果必须使用 AWS

如因架构限制必须使用 AWS：
1. 仅用于**低频 OTP**场景
2. 使用**纯英文** OTP 消息以降低成本
3. 实施**重试机制**应对送达失败
4. 提供**备选验证方式**（邮件、语音电话）
5. 明确告知用户 SMS 可能来自随机号码
6. 使用 **ap-northeast-2（首尔）** 区域以降低延迟
